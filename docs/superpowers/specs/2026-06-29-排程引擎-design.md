# 排程引擎（scheduler）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑥ 主動關懷的「排程引擎（Cron，借鏡 Hermes cron）」。
> 觸發對象（本刀）：⑦ 長期記憶骨幹 [長期記憶骨幹](2026-06-29-長期記憶骨幹-design.md)。

---

## 1. 背景與目標

長期記憶骨幹（`run_consolidation`）已完成，但需**手動跑 CLI**。本 spec 做一個**自建輕量排程器**，讓骨幹**每日自動對所有長者跑一次整理**；它同時是**主動關懷**未來要插入「定時問候／提醒」的共用引擎。

**完成後：** 啟動一個 worker，它每日到點自動枚舉所有長者 session、各跑一次骨幹整理。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* 輕量 `Scheduler`：持有 jobs、判斷「每日到點且當天未跑才執行」（時鐘注入、可測）、單一 job 例外不中斷其他。
* 一個 job：`build_consolidation_job` —— 對**所有 session** 跑 `run_consolidation`，單一 session 失敗不影響其他。
* `worker`：長跑迴圈 + CLI（`python -m kinsun.scheduler`）。
* `MemoryStore.sessions()`：枚舉所有 session。

### 2.2 不在範圍內（後續）
* 主動關懷的 job（問候／用藥提醒／失聯關心）——之後註冊到同一個 `Scheduler`。
* 持久化排程狀態（v1 用 in-memory 記「當天已跑」；重啟當天可能重跑一次，骨幹去重故無害）。
* cron 表達式、秒級精度、分散式多 worker。

## 3. 元件與檔案

```
src/kinsun/scheduler/
  __init__.py
  scheduler.py   # Job + Scheduler（run_due / 到期判斷）
  jobs.py        # build_consolidation_job
  worker.py      # build_scheduler + serve + main
  __main__.py    # 使 `python -m kinsun.scheduler` 可用
src/kinsun/memory/store.py   # 加 MemoryStore.sessions()
src/kinsun/config.py         # 加 consolidation_hour / scheduler_tick_seconds
```

### 3.1 排程器（scheduler.py）
```python
@dataclass(frozen=True)
class Job:
    name: str
    hour: int          # 每日觸發時（設定時區）
    minute: int
    run: Callable[[], None]

class Scheduler:
    def __init__(self, jobs: list[Job], clock: Callable[[], datetime]) -> None: ...
    def run_due(self) -> list[str]: ...   # 回傳本次執行的 job 名稱
```
* `run_due`：對每個 job，若「當天尚未跑」且「現在 ≥ 該 job 的時:分」→ 執行、記下當天日期、加入回傳清單。
* **錯誤隔離**：`job.run()` 拋例外 → 記錄並續跑其他 job；仍標記當天已跑（避免同日重試洗版）。
* `_is_due(job, now)`：`last_run[name] != now.date()` 且 `(now.hour, now.minute) >= (job.hour, job.minute)`。
* `last_run` 為 in-memory `dict[str, date]`。

### 3.2 整理 job（jobs.py）
```python
def build_consolidation_job(
    *, sessions: Callable[[], list[str]], run_one: Callable[[str], object],
    hour: int, minute: int = 0, name: str = "daily-consolidation",
) -> Job: ...
```
* `run()`：`for s in sessions(): run_one(s)`；**單一 session 失敗 → 記錄並續跑其他**。

### 3.3 worker（worker.py）
* `build_scheduler(settings, *, clock) -> Scheduler`：組真實元件（memory／extractor／embedder／兩 store），`run_one = run_consolidation(...)`，回 `Scheduler([job], clock)`。
* `serve(scheduler, *, tick_seconds)`：`while True: run_due(); sleep(tick)`（薄膠水、不單測）。
* `main()`：load settings → build → serve。
* `__main__.py`：`python -m kinsun.scheduler` 入口。

### 3.4 配套
* `MemoryStore.sessions() -> list[str]`、`SqliteMemoryStore`：`SELECT DISTINCT session_id FROM turns ORDER BY session_id`。
* `Settings.consolidation_hour: int`（預設 3）、`scheduler_tick_seconds: int`（預設 60）；`.env.example` 補上。

## 4. 資料流

```
worker 每 tick_seconds → scheduler.run_due()
  └ 若 03:00 後且當天未跑 → daily-consolidation
        └ for session in memory.sessions(): run_consolidation(session, …)
              └ 抽取（一次 LLM）→ 路由（事實→知識庫、情緒→向量庫）
```

---

## 5. 錯誤處理（可靠性）

* **單一 job 例外** → 記錄、續跑其他 job、排程器不崩。
* **單一 session 整理失敗** → 記錄、續跑其他 session。
* 排程屬非對話路徑；失敗只記錄，不影響線上對話。
* `sessions()` DB 例外 → 包成既有 `MemoryError`。

## 6. 測試策略（全本機，不需網路）

* `test_scheduler`：到點才觸發（FakeClock）；當天不重跑；跨日再跑；時間未到不跑；**單一 job 爆掉不影響其他**（兩 job 皆嘗試）。
* `test_scheduler_jobs`：`build_consolidation_job` 對每個 session 呼叫 `run_one`；**單一 session 失敗仍續跑其他**。
* `test_memory`（追加）：`sessions()` 回 distinct、排序。
* `test_config`：新預設值。
* `worker`（`build_scheduler`/`serve`/`main`）：wiring 與無窮迴圈，不單測；以 import／build smoke 驗證。

## 7. 已定預設值（列出供否決）

* **獨立 worker** 程序（`python -m kinsun.scheduler`），與 webhook 解耦。
* in-memory 記「當天已跑」（不另存狀態 DB）。
* 每日單一時點 `consolidation_hour`（預設 3 點），分鐘預設 0。
* job 失敗仍標記當天已跑（一天一次，不同日重試）。
* 零新依賴。
