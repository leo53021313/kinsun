# 帳號綁定核心 v1（資料層 + 綁定生命週期）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 資料層、② 後端服務。
> 策略文件：[帳號綁定與知情同意設計](../../帳號綁定與知情同意設計.md)（本 spec 為其第一刀實作）。

---

## 1. 背景與目標

危急通知、用藥提醒、家屬端都需要「長者—家屬關聯」這份資料，但目前不存在。本 spec 做**帳號綁定子系統的第一塊（切片 A）**：資料層 + 綁定生命週期的**純後端核心**（不含 LINE/LIFF UI）。

**完成後：** 程式可建立長者檔、產生一次性邀請碼、兌換綁定（長者本人或其他家屬）、記錄知情同意、查詢家屬清單與權限——成為下游（用藥提醒／危急真實通知／家屬端）的地基。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* 5 實體 + SQLite repository（behind `AccountRepository` Protocol → 未來 `PgAccountRepository`）。
* 綁定生命週期 service：建檔、產生邀請、兌換綁定（含驗證）、知情同意、撤回、家屬清單、權限查詢。
* 1 長者 ── N 家屬。

### 2.2 不在範圍內（後續切片／治理層）
* LINE/LIFF UI（家屬端）、長者端綁定流程（webhook）、後端 HTTP API。
* 稽核日誌、加密、去識別化（資料治理層）。
* 換手機重綁、解除綁定的資料刪除流程、代理同意的法律佐證。
* **用藥提醒／危急真實通知**（讀本資料的下游功能）。

## 3. 資料模型（models.py）

```python
class Role(StrEnum):        PRIMARY="primary";   GUARDIAN="guardian"
class InviteRole(StrEnum):  ELDER="elder";       GUARDIAN="guardian"
class ConsentBy(StrEnum):   SELF="self";         PROXY="proxy"

@dataclass(frozen=True)
class Elder:          elder_id; name; line_user_id: str | None = None
class Guardian:       guardian_id; line_user_id; name
class ElderGuardian:  elder_id; guardian_id; role: Role; escalation_order: int; can_view_transcript: bool
class Consent:        elder_id; consent_by: ConsentBy; version: str; granted_at: float; revoked_at: float | None = None
class Invite:         code; elder_id; role: InviteRole; expires_at: float; max_attempts: int; attempts: int = 0; used_at: float | None = None
```

## 4. 儲存（repository.py）

`AccountRepository`（Protocol）+ `SqliteAccountRepository(db_path)`，表 `elders / guardians / elder_guardians / consents / invites`。方法（upsert 用 `INSERT OR REPLACE`）：

```
save_elder / get_elder(elder_id) -> Elder | None
save_guardian / get_guardian_by_line(line_user_id) -> Guardian | None
save_elder_guardian / get_elder_guardian(elder_id, guardian_id) -> ElderGuardian | None
                    / list_elder_guardians(elder_id) -> list[ElderGuardian]
save_consent / get_consent(elder_id) -> Consent | None
save_invite / get_invite(code) -> Invite | None
```
DB 例外包成 `AccountError`。

## 5. 綁定生命週期（service.py）

`AccountService(repo, *, clock, new_id, new_code, ttl_hours=24, max_attempts=5)`
（`clock: () -> datetime`、`new_id: () -> str`、`new_code: () -> str` 皆**注入**以利 TDD；常數 `CONSENT_VERSION="1.0"`）

| 操作 | 行為 |
|------|------|
| `create_elder(guardian_line_id, guardian_name, elder_name) -> Elder` | 取得或建立 Guardian（依 line_user_id）→ 建 Elder → 建 `ElderGuardian(role=PRIMARY, escalation_order=1, can_view_transcript=True)`，回 Elder |
| `generate_invite(elder_id, role) -> Invite` | `code=new_code()`、`expires_at=clock()+ttl_hours`、`max_attempts`，存入，回 Invite |
| `redeem_invite(code, line_user_id, *, consent_by) -> None` | 驗證 → 綁定 → 記同意（見下） |
| `revoke_consent(elder_id) -> None` | 取 Consent → 設 `revoked_at=clock()` 存回 |
| `guardians_of(elder_id) -> list[ElderGuardian]` | 依 `escalation_order` 排序，**供危急通知升級順序** |
| `can_view_transcript(elder_id, guardian_id) -> bool` | 取 ElderGuardian → 回 `can_view_transcript`（無關聯 → False） |

### 5.1 `redeem_invite` 驗證與綁定
驗證（任一失敗 → `raise InviteError(reason)`）：
* `not_found`：查無 code。
* `used`：`used_at` 已設。
* `expired`：`clock() > expires_at`。
* `too_many_attempts`：`attempts >= max_attempts`。

通過後：
* `attempts += 1`、`used_at = clock()` 存回 invite。
* **role=ELDER**：`save_elder(elder with line_user_id=line_user_id)`（長者本人綁定）。
* **role=GUARDIAN**：取得或建 Guardian → `save_elder_guardian(role=GUARDIAN, escalation_order=max+1, can_view_transcript=False)`。
* `save_consent(Consent(elder_id, consent_by, CONSENT_VERSION, granted_at=clock(), revoked_at=None))`。

> 失敗（如 expired）時亦 `attempts += 1` 存回，以利「嘗試上限」防爆破。

## 6. 錯誤處理

* `AccountError`（DB／一般）、`InviteError(reason)`（兌換失敗，reason ∈ not_found/expired/used/too_many_attempts）。
* 純後端、非對話路徑；錯誤直接拋給呼叫端（未來 API/webhook 層轉成使用者訊息）。
