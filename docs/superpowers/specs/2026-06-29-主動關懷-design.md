# 主動關懷 v1（定時問候 + 失聯關心）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑥ 主動關懷（Cron 主動觸發 Care Agent：問候／失聯關心）。
> 依賴：排程引擎 [排程引擎](2026-06-29-排程引擎-design.md)、長期記憶（知識／向量／骨幹）、短期記憶。

---

## 1. 背景與目標

系統目前只會「被動回答」。本 spec 讓金孫**主動關懷**：排程器定時觸發、**主動推播**溫暖訊息給長者。

**完成後：** 每日定時對每位長者推一則個人化問候；長者太久沒互動時主動關心。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* **主動推播**：`LineMessenger.push_text(user_id, text)`（LINE push，不靠回覆 token）。
* **主動組訊息**：`CareAgent.proactive(session_id, intent)`——帶記憶（知識＋情緒）用 LLM 生成，寫回短期記憶。
* **定時問候 job**：每日到點，對每位長者組訊息並推播。
* **失聯關心 job**：每日到點，長者最後互動超過門檻者，主動關心。
* `MemoryStore.last_active(session_id)`：最後一次**長者**發話時戳。

### 2.2 不在範圍內（後續）
* **用藥／回診提醒**：需結構化用藥時間／回診日（資料層／帳號綁定）。
* 危急偵測的行為訊號 C（與失聯關心相關，但走危急升級鏈，屬危急模組）。
* 推播頻率精細控制（v1 靠「每日 job 一天一次」自然限流）。

## 3. 元件與檔案

```
src/kinsun/proactive/
  __init__.py
  jobs.py        # GREETING_INTENT / INACTIVITY_INTENT + build_greeting_job + build_inactivity_job
src/kinsun/agent.py                    # + CareAgent.proactive()
src/kinsun/channels/line/messenger.py  # + LineMessenger.push_text + LineApiMessenger.push_text
src/kinsun/memory/store.py             # + MemoryStore.last_active()
src/kinsun/scheduler/worker.py         # 註冊問候 / 失聯 job
src/kinsun/config.py                   # greeting_hour / inactivity_hour / inactivity_days
```

### 3.1 主動推播（messenger.py）
* `LineMessenger`（Protocol）加 `push_text(user_id, text) -> None`。
* `LineApiMessenger.push_text`：`MessagingApi.push_message(PushMessageRequest(to=user_id, messages=[TextMessage(text)]))`。不單測（包 SDK）。

### 3.2 主動組訊息（agent.py）
```python
class CareAgent:
    def proactive(self, session_id: str, intent: str) -> str: ...
```
* `system_prompt = SYSTEM_PROMPT + context.recall(session_id, intent)`；以 `intent` 為語意檢索查詢。
* 送 `history + [指令訊息]`（指令描述主動關懷意圖，**屬系統提示、非長者發話**）給 LLM。
* 把生成的回覆**寫回短期記憶**（assistant turn，**不寫指令**），回傳訊息。

### 3.3 最後互動（memory/store.py）
* `last_active(session_id) -> float | None`：`SELECT MAX(created_at) FROM turns WHERE session_id=? AND role='user'`；無則 `None`。
* **只算長者（user）發話**——我們的主動推播（assistant）不算，避免洗掉失聯訊號。

### 3.4 主動關懷 job（proactive/jobs.py）
```python
def build_greeting_job(*, sessions, greet_one, hour, minute=0, name="daily-greeting") -> Job
def build_inactivity_job(*, sessions, last_active, clock, threshold_seconds, care_one,
                         hour, minute=0, name="inactivity-care") -> Job
```
* 問候 job：`for s in sessions(): greet_one(s)`，**單一失敗不影響其他**。
* 失聯 job：`for s in sessions(): if last_active(s) 非 None 且 now - last_active(s) >= 門檻: care_one(s)`，錯誤隔離；`last_active` 為 `None`（從未發話）→ 跳過。

### 3.5 worker 註冊
`build_scheduler` 額外組 `CareAgent`、`LineApiMessenger`，註冊：
* `greet_one(s) = messenger.push_text(s, agent.proactive(s, GREETING_INTENT))`
* `care_one(s) = messenger.push_text(s, agent.proactive(s, INACTIVITY_INTENT))`
* 門檻 `threshold_seconds = inactivity_days * 86400`。

---

## 4. 流程（插進現有 Scheduler）

```
排程器每日：
  03:00 → 骨幹整理（已做）
  08:00 → 問候 job：for 長者: push(agent.proactive(早安問候))
  10:00 → 失聯 job：for 長者: if 超過 N 天沒發話: push(agent.proactive(主動關心))
```

`proactive` 寫回短期記憶 → 對話連貫；每日 job → 一天最多一次，天然限流。

---

## 5. 錯誤處理（可靠性）

* 單一長者問候／關心失敗 → 記錄並續跑其他（job 內錯誤隔離）。
* job 整體例外 → 排程器已有錯誤隔離，不崩。
* `last_active` DB 例外 → 包成既有 `MemoryError`。
* 非對話路徑；失敗只記錄。

## 6. 測試策略（全本機，不需網路）

* `test_memory`（追加）：`last_active` 回最後**長者**發話時戳；只 assistant／無 turn → `None`。
* `test_agent`（追加）：`proactive` 帶記憶生成（system prompt 含 context）、指令含 intent、回覆寫回短期記憶（不寫指令）、回傳訊息。
* `test_proactive_jobs`：問候 job 對每位長者呼叫 `greet_one`、錯誤隔離；失聯 job 只對「超門檻且非 None」者呼叫 `care_one`、None／新近者跳過、錯誤隔離。
* `test_config`：新預設值。
* `messenger.push_text`／`worker`：不單測，以 fake／smoke 驗證（`build_scheduler` 回含 3 個 job 的 Scheduler）。

## 7. 已定預設值（列出供否決）

* `GREETING_HOUR`(8)、`INACTIVITY_HOUR`(10)、`INACTIVITY_DAYS`(2)。
* `proactive` 每次一個 LLM 呼叫；訊息寫回短期記憶。
* 失聯用「最後 user 發話」+ 每日 job 限流。
* LINE **push 計入官方帳號訊息配額**（reply 免費、push 計費）；MVP 無妨。
* 零新依賴。
