# 關鍵健康事實每輪必帶（固定健康查詢增補）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶/資料層；亦實現 [知識圖譜拆解決策](../../知識圖譜拆解決策.md) 的用途 ②。
> 前置：[記憶層雲端遷移](2026-06-29-記憶層雲端遷移-design.md)、[照護抽取 prompt 接線](2026-06-29-照護抽取prompt接線-design.md)、[Mem0 新演算法對齊與清死設定](2026-06-29-mem0新演算法對齊與清死設定-design.md)。

---

## 1. 背景與目標

長者的用藥、慢性病、過敏等**穩定關鍵事實**，若只靠「以當下 user 文字做語意檢索」，當話題與健康無關時不會浮現 → 金孫可能在不知情下回應，對照護產品是正確性風險。

實測 mem0 2.0.10：`add()` 抽出的每筆事實**共用同一份 metadata**（`memory/main.py:976` 對每筆 deepcopy 同一 metadata），**無法 per-fact 分類**，故「以 `category` 過濾 get_all」不可行（除非自建抽取分類，不值得）。

**採用固定健康查詢增補**：每輪 recall 除了 user 文字檢索，**固定再用一組健康關鍵詞檢索一次**，raw 層合併去重後一起進 prompt。健康事實因此每輪都有機會出現，與當下話題無關亦然。

**完成後：** 對話時 system prompt 穩定帶入長者的健康相關長期記憶（上限 N 筆）；改動封裝在 `Mem0LongTermStore` 內，其餘元件不變；全程 fail-safe。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* `Mem0LongTermStore` 新增固定健康查詢增補：`search` 內跑 user 文字檢索 + `HEALTH_QUERY` 檢索，依 mem0 `id` 去重（user 優先）、健康事實上限 `health_top_k`，合併格式化。
* 新增常數 `HEALTH_QUERY`、建構參數 `health_top_k`（預設 3）、私有 `_search_raw`／去重輔助。
* 對應單元測試（離線，fake mem0）。

### 2.2 不在範圍內（YAGNI）
* **結構化 category 分類**：Mem0 無法 per-fact，硬做要自建抽取；不做。
* **個人化健康查詢**：本案用固定關鍵詞；未來再議。
* **新環境變數**：`health_top_k` 為建構參數（預設 3），不進 env／`Settings`。
* `LongTermStore` Protocol、`MemoryContext`、`CareAgent`、`FakeLongTermStore` 一律不動。

## 3. 資料流

```
CareAgent.handle → MemoryContext.recall(session_id, user_text)
  → Mem0LongTermStore.search(session_id, user_text):
       A = _search_raw(user_text,    session_id, top_k)          # 當下相關
       H = _search_raw(HEALTH_QUERY,  session_id, health_top_k)  # 健康必帶
       merged = dedup_by_id(A + H)   # 同 id 只留一次，A 優先
  → _format_memories_for_prompt({"results": merged}) → 注入 system prompt
```

`HEALTH_QUERY = "用藥 慢性病 過敏 回診 健康狀況"`（固定）。`health_top_k` 預設 3。

        # key = item["id"] or item["memory"] or item["text"]；保留首次出現
        ...

    def search(self, session_id: str, query: str, *, top_k: int | None = None) -> str:
        user_items = self._search_raw(query, session_id, top_k or self._top_k)
        health_items = self._search_raw(HEALTH_QUERY, session_id, self._health_top_k)

* `add`、`_format_memories_for_prompt`、`_PREFIX` 不變。
* `search` 對外簽名與回傳型別（`-> str`）不變 → Protocol／呼叫端零變更。
* `app.py`／`worker.py` 仍以 `Mem0LongTermStore(build_mem0_memory(settings), top_k=settings.longterm_top_k)` 建構（`health_top_k` 取預設 3）。

## 5. 錯誤處理（fail-safe）

* 兩次檢索各自在 `_search_raw` 內 try/except → 任一失敗回 `[]`、記 warning、不上拋。
* 健康檢索失敗仍回 user 結果；user 檢索失敗仍回健康結果；皆失敗回 `""`（同現行「退化為無記憶、不中斷對話」）。

---

## 6. 測試策略（單元全離線）

於 [tests/test_longterm_store.py](../../../tests/test_longterm_store.py) 以「依 query 回不同 results 的 fake mem0」新增：
* **健康必帶**：user 查詢「今天天氣」回非健康事實、`HEALTH_QUERY` 回「有高血壓」→ `search` 輸出**同時含兩者**（健康與話題無關仍出現）。
* **去重**：user 與健康回相同 `id` → 該事實只出現一次。
* **上限與雙檢索**：記錄每次 `search` 的 `limit` → 確認以 `top_k` 查 user、以 `health_top_k` 查 `HEALTH_QUERY`（兩次都發生、限額正確）。
* **健康檢索失敗隔離**：`HEALTH_QUERY` 檢索丟例外 → 仍回 user 結果、不崩。
* 既有 `test_search_returns_formatted_string`、`test_search_failsafe_returns_blank` 在新設計下仍通過（向後相容）。

## 7. 已知取捨（列出供否決）

* **每輪兩次 `mem0.search`**：latency／成本約加倍；LINE 非即時、一來一往，可接受。
* **固定健康關鍵詞、非個人化**：簡單可預期；代價是對特殊健康詞彙可能漏接，未來可調 `HEALTH_QUERY` 或改個人化。
* **不做 per-fact 分類**：受 Mem0 per-batch metadata 限制 + YAGNI；關係/分類感知交由 v1.1 多訊號檢索。
* **健康事實實際品質待雲端驗證**：離線測證「機制正確（健康查詢一定發生、結果被帶入、去重、fail-safe）」；真實檢索品質待金鑰冒煙。
