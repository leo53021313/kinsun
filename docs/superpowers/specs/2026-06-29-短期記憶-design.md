# 短期記憶（今日對話上下文）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶／資料層的「短期記憶（今日對話上下文）」。
> 銜接第一刀 [端到端語音薄切片](2026-06-28-端到端語音薄切片-design.md)：把無狀態的 Care Agent 升級為「記得今天聊過什麼」。

---

## 1. 背景與目標

第一刀的 Care Agent 是**無狀態**的：長者每講一句都當第一次見面。對「陪伴」產品這是最刺眼的缺口，也是 ⑦ 記憶層的第一塊地基——後續長期記憶、主動關懷、危急偵測的行為訊號都依賴它。

**完成後要做到：** 長者在同一天內多輪對話時，金孫能接續前面的上下文回應。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* 每輪對話（長者說的、金孫回的）持久化進 **SQLite**。
* 每次回覆前載入「**今日**稍早對話」（上限最近 N 輪），與本輪一起餵 Gemini。
* `LLMClient` 介面演進為吃**多輪訊息串列**。
* session 以 **LINE userId** 為 key。

### 2.2 不在範圍內（後續子專案）
* 每日批次抽取短期 → 長期記憶（Cron＋長期記憶層）。
* 摘要／壓縮、跨裝置彙整、長者檔案（知識圖譜）。
* 舊資料清除（保留給未來 Cron 抽取後處理）。

## 3. 元件與檔案

### 3.1 新增
```
src/kinsun/memory/
  __init__.py
  store.py     # MemoryStore(Protocol) + MemoryError + SqliteMemoryStore
tests/
  test_memory.py
```

### 3.2 介面演進（本模組核心改動）
於 [llm.py](../../../src/kinsun/llm.py) 新增共用型別，並把 `generate` 改吃訊息串列：

```python
@dataclass(frozen=True)
class Message:
    role: str   # "user" | "assistant"
    text: str

class LLMClient(Protocol):
    def generate(self, *, system_prompt: str, messages: list[Message]) -> str: ...
```

`GeminiClient` 把 `messages` 映射為 Gemini 原生 contents（`assistant` → `model`）。映射抽成純函式 `_to_contents(messages) -> list`，可獨立單測。

### 3.3 記憶介面
```python
class MemoryError(Exception): ...

class MemoryStore(Protocol):
    def append(self, session_id: str, message: Message) -> None: ...
    def recent(self, session_id: str) -> list[Message]: ...   # 只回今日、依時間序、上限 N 輪
```

`SqliteMemoryStore(db_path: str, clock: Callable[[], datetime], max_turns: int)`：
* 表 `turns(id INTEGER PK, session_id TEXT, role TEXT, text TEXT, created_at REAL)`，`created_at` 存 epoch 秒。
* 單一連線、`check_same_thread=False`（MVP 低流量足夠；高併發再議）。
* `append`：用注入的 `clock()` 取得當下時間戳寫入。
* `recent`：以 `clock()` 算出「**今日 00:00（設定時區，預設 Asia/Taipei）**」的 epoch 下界，撈該 session 當天的列、依 `created_at` 排序、取最後 `max_turns` 輪，回傳 `list[Message]`。
* DB I/O 例外包成 `MemoryError`。

> **時鐘注入**是為了 TDD 能用固定時間驗證「昨天的不該被載入」「上限生效」。

## 4. 資料流（Care Agent 長出記憶）

```
CareAgent.handle(session_id, user_text):
    history = memory.recent(session_id)               # 今日稍早（可能空）
    user_msg = Message("user", user_text)
    reply = llm.generate(system_prompt=SYSTEM_PROMPT, messages=[*history, user_msg])
    memory.append(session_id, user_msg)               # 成功後才寫
    memory.append(session_id, Message("assistant", reply))
    return reply
```

session 串接：`webhook(event.source.user_id)` → `pipeline.process(audio, session_id=...)` → `agent.handle(session_id, text)`。

## 7. 錯誤處理

* **記憶壞掉不該讓對話整個掛掉**：`recent()` 載入失敗時，記錄錯誤並退化為「無歷史」繼續回覆（fail-safe，呼應主軸的可靠性傾向）。
* `append()` 失敗記錄錯誤，不阻斷回覆（已回給長者為優先）。
* LLM 失敗時**不寫入該輪**，避免孤兒 user turn；webhook 走既有 fallback。
* 所有 DB 例外以可辨識的 `MemoryError` 呈現（呼應 [AGENTS.md](../../../AGENTS.md) 錯誤處理）。

## 8. 測試策略（全本機，不需 GPU／憑證）

* `test_memory`：append 後 recent 依序回傳；昨天的列被排除（固定 clock）；超過 `max_turns` 只回最後 N 輪；不同 session 互相隔離；`recent` 對空 session 回空陣列。
* `test_llm`：`_to_contents()` 角色映射（`assistant`→`model`、`user`→`user`）；更新 FakeLLM 簽章。
* `test_agent`：`handle` 會載入歷史、把「歷史＋本輪」交給 LLM、成功後寫回兩輪（用 spy memory + spy llm）。
* `test_pipeline`／`test_webhook`：session_id 正確串接；source 無 user_id 時退化為 `"unknown"` 仍可回覆。

整合測試（連真 Gemini／DGX）仍為手動，不在本模組 CI。

---

## 9. 已定預設值（列出供否決）

* 「今日」= 設定時區（預設 Asia/Taipei）的日曆當天，非滾動 24 小時。
* 歷史上限 20 輪（user+assistant 合計輪數的最後 N 筆）。
* 單一 SQLite 連線、`check_same_thread=False`，適用 MVP 低流量。
* 舊資料不刪，僅不載入（留給未來 Cron）。
