# 記憶層雲端遷移（Mem0-g + Supabase + Neo4j）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶／資料層。
> 取代既有自建 SQLite 記憶堆疊，落地架構圖標示的 **長期記憶骨幹（Mem0-g）／知識圖譜（圖資料庫）／向量庫（pgvector）**。
> 銜接並取代：[知識圖譜 v1](2026-06-29-知識圖譜v1-design.md)、[向量庫 episodic v1](2026-06-29-向量庫episodic-design.md)、[長期記憶骨幹](2026-06-29-長期記憶骨幹-design.md)。
> 相關安全核心：[對話安全與資料正確性-待議](../../對話安全與資料正確性-待議.md)（provenance）。

---

## 1. 背景與目標

目前記憶層全部是自建 SQLite：短期記憶（`SqliteMemoryStore`）、知識圖譜（`SqliteFactStore`）、向量庫（`SqliteVectorStore` + 純 Python 餘弦）、自建抽取（`ConsolidationExtractor`）。帳號綁定亦用 `SqliteAccountRepository`。

本 spec 將記憶層技術選型**對齊架構圖**並整體上雲：

* 長期記憶 backbone 改用 **Mem0**（graph 模式，即架構圖「Mem0-g」），由 Mem0 統一處理抽取、embedding、向量檢索、去重／衝突解決，並自動長出實體關係圖。
* 向量庫 → **Supabase（Postgres + pgvector）**；知識圖譜 → **Neo4j Aura**；抽取與 embedding 的 LLM → **Gemini**（沿用現有 key）。
* 短期記憶與帳號綁定 → **Supabase Postgres**。
* **移除專案內所有 SQLite 相依**（含帳號綁定）。

**完成後：** 記憶讀寫全部走雲端共用儲存（webhook server 與 scheduler worker 可跨程序／跨機器共享狀態）；長期記憶具備圖關係與成熟檢索；對話安全的 **provenance** 紀律以 Mem0 metadata + 自訂抽取 prompt 維持。

### 1.1 已鎖定的選型決策（brainstorming 結論）

| 項目 | 決策 |
|------|------|
| 框架 | **Mem0-g**（graph 模式）；非 Letta |
| 向量後端 | **Supabase**（Mem0 `vector_store.provider = "supabase"`，pgvector） |
| 圖後端 | **Neo4j Aura** 免費 tier（Mem0 `graph_store.provider = "neo4j"`） |
| LLM／embedder | **Gemini**（Mem0 `llm`／`embedder` provider = gemini） |
| 整合方式 | **方案 A：Mem0 接管**（`infer=True`，自動抽取＋圖關係＋向量） |
| provenance 粒度 | **依來源**（長者對話批次＝`self_claimed`；家屬確認＝`family_confirmed`），非逐筆 |
| SQLite | **全部移除**，含帳號綁定 |
| 舊資料 | **直接捨棄、不搬移**（dev 資料） |
| 本機環境 | **不裝本機 DB**；dev／正式皆連雲端；單元測試離線用 fake |

## 2. 範圍（Scope）

### 2.1 在範圍內
* 新增薄介面 `LongTermStore`（Protocol）+ `Mem0LongTermStore` 實作，包住 Mem0。

* 帳號綁定 `PgAccountRepository`（Supabase Postgres）取代 `SqliteAccountRepository`。

* 測試重整：單元測試改注入記憶體 fake；新增 opt-in 整合測試。

* 帳號綁定接入對話流程的「綁定閘門」（仍是既有缺口，另案）。

* 危急通知、用藥提醒等其他模組。
* 舊 SQLite 資料搬移腳本（已決議捨棄）。
* 進階檢索調校（重排序、時間衰減）與「穩定事實每輪必帶」強化（見 §7 已知取捨）。

### 3.1 長期記憶薄介面（longterm/store.py）

```python
class LongTermStore(Protocol):
    def add(self, session_id: str, messages: list[Message], *, provenance: str = "self_claimed") -> None: ...
    def search(self, session_id: str, query: str, *, top_k: int = 5) -> str: ...

    def __init__(self, memory, *, top_k: int = 5) -> None: ...   # memory: mem0.Memory
    def add(self, session_id, messages, *, provenance="self_claimed") -> None:
        # messages → [{"role": m.role, "content": m.text}]；呼叫 memory.add(..., user_id=session_id,
        #   metadata={"provenance": provenance})，infer=True（預設）
    def search(self, session_id, query, *, top_k=None) -> str:
        # memory.search(query, user_id=session_id, <limit/top_k=...>) → _format_memories_for_prompt
        # 任一例外 → 記錄並回 ""（fail-safe，不中斷對話）
```

* `_format_memories_for_prompt(results) -> str`：無結果回 `""`；有則回前綴＋逐筆列出，依 metadata 的 provenance 加中文標註（例 `（長者自述）`）。可選擇附帶 graph relations（`results["relations"]`）作為關係提示。格式維持與現行注入字串相近的風格。
* `user_id = session_id`（即 LINE userId；帳號綁定接入後再改為 `elder_id`，屬另案）。

### 3.2 provenance 與自訂抽取 prompt（longterm/provenance.py）

```python
SELF_CLAIMED = "self_claimed"        # 長者自述
INFERRED = "inferred"                # 推測
FAMILY_CONFIRMED = "family_confirmed" # 家屬確認

LABELS = {SELF_CLAIMED: "長者自述", INFERRED: "推測", FAMILY_CONFIRMED: "家屬確認"}

CUSTOM_FACT_EXTRACTION_PROMPT = """..."""  # 植入安全紀律：
# - 健康宣稱若僅長者自述、未經確認，不可當成已確診的醫療事實。
# - 不替長者下醫療診斷；對認知退化者的前後矛盾不爭辯、不當事實。
# - 只抽穩定、對照護有意義的事實與有情緒意義的片段。
# - 輸出格式遵循 Mem0 要求的 {"facts": [...]}。
```

> provenance 以 **add() 來源粒度**寫入 metadata：長者每日對話批次預設 `self_claimed`；未來家屬確認流程寫 `family_confirmed`。此粒度正好對應真實情況（長者自述本即未確認）。

### 3.3 Mem0 組裝（mem0_factory.py）

```python
def build_mem0_memory(settings):
    config = {
        "llm": {"provider": "gemini",
                "config": {"model": settings.gemini_model, "api_key": settings.gemini_api_key}},
        "embedder": {"provider": "gemini",
                     "config": {"model": settings.embedding_model}},
        "vector_store": {"provider": "supabase",
                         "config": {"connection_string": settings.database_url,
                                    "collection_name": "kinsun_memories",
                                    "index_method": "hnsw",
                                    "index_measure": "cosine_distance"}},
        "graph_store": {"provider": "neo4j",
                        "config": {"url": settings.neo4j_uri,
                                   "username": settings.neo4j_username,
                                   "password": settings.neo4j_password}},
    }
    return Memory.from_config(config)
```

> **spike 需確認的設定細節**（不臆測，實作時查證 Mem0 當前版本）：
> 1. `search` 的取回數參數名（`limit` 或 `top_k`）與回傳結構（`results`/`relations` 欄位）。
> 2. Gemini embedder 模型字串（文件示例 `models/gemini-embedding-001`）與 **embedding 維度**，並讓 `vector_store` 的 `embedding_model_dims` 與之一致。
> 3. Gemini LLM provider 是否需額外環境變數（`GOOGLE_API_KEY`）。
> 4. 啟用 graph 模式所需的額外安裝項（如 `mem0ai[graph]`）。

### 3.4 短期記憶（memory/store.py）

`MemoryStore` Protocol **不變**（`append`／`recent`／`sessions`／`last_active`）。`PgMemoryStore`：

* Postgres 表 `turns(id BIGSERIAL PK, session_id TEXT, role TEXT, text TEXT, created_at DOUBLE PRECISION)`，索引 `(session_id, created_at)`。
* 語意與 `SqliteMemoryStore` 一致：`recent` 只回「今日」（設定時區日界）且至多 `max_turns` 筆；`last_active` 只看 `role='user'`。
* 全部用**參數化查詢**（防 SQL injection）；DB 例外包成 `MemoryError`。
* 連線經 `kinsun.db` 的共用 psycopg 連線／連線池。

### 3.5 帳號綁定（accounts/repository.py）

`AccountRepository` Protocol、`AccountService`、`models.py` **不變**。`PgAccountRepository`：

* Postgres 5 表：`elders`、`guardians`、`elder_guardians`、`consents`、`invites`，欄位對應既有 SQLite schema（型別改 Postgres：`TEXT`/`BIGINT`/`DOUBLE PRECISION`/`BOOLEAN`）。
* 主鍵與唯一約束維持：`consents` 以 `elder_id` 為 PK；`invites` 以 `code` 為 PK；`guardians.line_user_id` UNIQUE；`elder_guardians (elder_id, guardian_id)` PK。
* 參數化查詢；DB 例外包成 `AccountError`。

### 3.6 連線工廠（db.py）

* `build_pool(database_url)` / `connect(database_url)`：以 psycopg（v3）建立連線（MVP 低流量可用簡單連線池）。
* 建表 DDL（`turns` + 帳號 5 表）以 `CREATE TABLE IF NOT EXISTS` 冪等建立；Mem0 自管其向量表與 Neo4j。
* 不寫死任何 secret；全走 `DATABASE_URL`/`NEO4J_*` 環境變數（位置無關）。

## 6. 錯誤處理（可靠性）

* **線上讀路徑 fail-safe**：`Mem0LongTermStore.search`、`PgMemoryStore.recent/last_active` 任一例外 → 記錄並退化（回 `""`／`[]`／`None`），**絕不中斷對話**（沿用現行原則）。
* **線上寫**：`PgMemoryStore.append` 失敗記錄但不阻斷回覆。
* **批次寫**（consolidation）：屬非對話路徑；單一 session 失敗由 scheduler 既有的 per-session 隔離吸收，不影響其他 session。
* 自訂例外沿用：`MemoryError`、`AccountError`；Mem0 例外於 `search` 內吞掉、於 `add`（批次）可上拋給 job 記錄。
* secret 全走環境變數；查詢全參數化（防 SQL injection）。

## 7. 已知取捨與風險

1. **語意檢索 vs 穩定事實**：Mem0 `search` 為語意式，穩定事實（用藥／慢性病）未必每輪都浮現（現行 `KnowledgeRecaller` 是「全撈」）。v1 接受，靠 `top_k` 調校；未來可加「關鍵事實每輪必帶」（依 metadata 過濾 `get_all`）強化。
2. **provenance 粒度下降**：由逐筆 → 依來源批次。已評估：長者自述本即未確認，粒度貼合真實，且相容 Mem0 metadata 模型。
3. **無法離線跑 app**：移除全部 SQLite 後，實際啟動一定要雲端 key（Supabase + Neo4j）。單元測試仍離線（fake）。7 人團隊每人需雲端存取。
4. **新依賴重量**：`mem0ai` 相依較多；ARM64 相容必須 spike 驗證，過不了就回報、不硬幹。
5. **測試改寫成本**：移除 SQLite store 後，store 層測試移除或轉整合，邏輯測試改注入 fake。

## 9. 測試策略

> 原則維持：**單元測試全離線、不需網路／雲端 key**；雲端實作以 opt-in 整合測試覆蓋。

### 9.1 單元測試（預設 `pytest` 跑，離線）
* 新增記憶體 fake：`FakeLongTermStore`（記 add 呼叫、回固定 search 字串）、`FakeMemoryStore`、`FakeAccountRepository`（純 dict）。
* `test_memory_context`：`MemoryContext.recall` 委派 `long_term.search`；fail-safe（search 拋錯 → `""`）。
* `test_longterm_store`：`_format_memories_for_prompt`（無結果→`""`、provenance 標註、relations 呈現）；`Mem0LongTermStore.search` 以 fake mem0 物件驗 fail-safe 與參數傳遞；`add` 的 messages→dict 與 metadata 映射。
* `test_longterm_consolidation`：`run_consolidation` 用 fake short_term + fake long_term → 驗證讀今日、以 `self_claimed` 寫入、計數。
* `test_accounts_service`：沿用，改注入 `FakeAccountRepository`（不碰 DB）。
* `test_agent`／`test_pipeline`／`test_webhook`／`test_scheduler*`／`test_proactive*`：改用 fake 注入，行為不變。
* 移除：`test_knowledge_*`、`test_episodic_*`、`test_longterm_extractor`、SQLite 版 `test_memory`／`test_accounts_repository`（轉整合）。

### 9.2 整合測試（opt-in，`KINSUN_IT=1` + 真 keys 才跑）
* `PgMemoryStore`：append/recent（今日界線、max_turns）、sessions、last_active 對真 Supabase。
* `PgAccountRepository`：5 表 CRUD、約束、去重對真 Supabase。
* `Mem0LongTermStore`：add→search round-trip 對真 Supabase + Neo4j（小集合冒煙）。
* CI 預設略過；本地或專屬環境手動啟用。

## 10. 已定預設值（列出供否決）

* `LongTermStore.search` top_k 預設 5（`LONGTERM_TOP_K`）。
* Mem0 `user_id = session_id`（= LINE userId）；帳號綁定接入後再改 `elder_id`。
* Mem0 整合採方案 A（`infer=True`）；provenance 走 metadata、來源粒度。
* 向量 collection 名 `kinsun_memories`；index `hnsw` / `cosine_distance`。
* 短期＋帳號＋向量同一個 Supabase Postgres（`DATABASE_URL`）；圖另一個 Neo4j Aura。
* 舊 SQLite 資料捨棄、不搬移。
* 單元測試一律離線 fake；雲端走 opt-in 整合測試。
* 帳號綁定只換儲存實作，不在本案接入對話流程。
