# 長期記憶骨幹（consolidation）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶／資料層的「長期記憶骨幹（抽取·路由·檢索·更新）」。
> 銜接：[知識圖譜 v1](2026-06-29-知識圖譜v1-design.md)、[向量庫 episodic v1](2026-06-29-向量庫episodic-design.md)。

---

## 1. 背景與目標

長期記憶的兩塊「儲存」（知識圖譜、向量庫）已完成，但**每日抽取是兩條獨立批次、兩個 CLI**（`kinsun.knowledge.batch`、`kinsun.episodic.batch`），各自對同一段對話呼叫一次 LLM。本 spec 做「骨幹」的**寫入側**：**一次合併抽取 → 路由**，收斂成單一批次與 CLI。

> 讀取側的統一（知識＋情緒）已由 [MemoryContext](../../../src/kinsun/recall.py) 完成；本 spec 補上寫入側的統一。

**完成後：** 一個 CLI 跑一次，讀今日對話一次、呼叫 LLM 一次，產出事實＋情緒，分別路由進知識庫與向量庫。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* `ConsolidationExtractor`：**一次 LLM 呼叫**抽取 `{facts, episodes}`。
* `run_consolidation`：路由——事實 → 知識庫（`FactStore`）、情緒 → embed → 向量庫（`VectorStore`）。
* 單一 CLI：`python -m kinsun.longterm.consolidation <session_id>`。
* **移除**被取代的 `knowledge/extractor.py`、`knowledge/batch.py`、`episodic/extractor.py`、`episodic/batch.py` 及其測試。

### 2.2 不在範圍內（後續）
* Cron 排程（切片②；本批次仍手動/CLI 觸發）。
* 記憶「更新」（合併同一事實的新證據、可信度升級）——v1 仍靠去重，不做版本化。
* 線上 DB（store 介面不變）。

## 3. 元件與檔案

```
src/kinsun/longterm/
  __init__.py
  extractor.py       # Consolidation + ConsolidationExtractor + _parse_consolidation（純函式）
  consolidation.py   # ConsolidationResult + run_consolidation + main（CLI）
```

### 3.1 抽取（extractor.py）
```python
@dataclass(frozen=True)
class Consolidation:
    facts: list[Fact]      # 重用 kinsun.knowledge.facts.Fact
    episodes: list[str]

class ConsolidationExtractor:
    def __init__(self, llm: LLMClient) -> None: ...
    def extract(self, session_id: str, messages: list[Message]) -> Consolidation: ...
```
* 一次 LLM 呼叫，要求輸出 JSON 物件 `{"facts": [{category, content, provenance, confidence}], "episodes": ["…"]}`。
* `_parse_consolidation(session_id, raw) -> Consolidation`：純函式；解析物件、分別解析 facts／episodes；未知 category→`OTHER`、未知 provenance→`INFERRED`、clamp confidence、跳過缺 content／非字串；**壞 JSON / 非物件 → `Consolidation([], [])`**（fail-safe）。
* 抽取指令明示：健康宣稱僅自述 → `self_claimed`，不可當已確認。
* `messages` 為空或 LLM 失敗 → 空 `Consolidation`。

### 3.2 路由（consolidation.py）
```python
@dataclass(frozen=True)
class ConsolidationResult:
    facts_stored: int
    episodes_stored: int

def run_consolidation(
    session_id: str, *, short_term: MemoryStore, extractor: ConsolidationExtractor,
    embedder: Embedder, fact_store: FactStore, vector_store: VectorStore,
) -> ConsolidationResult: ...
```
* 讀今日短期 → `extractor.extract` → 事實逐筆 `fact_store.add`；情緒逐段 `embedder.embed`（**失敗跳過該段**）→ `vector_store.add`。
* `facts_stored = len(facts)`（嘗試數；知識庫以去重忽略重複）、`episodes_stored` = 成功 embed 並寫入數。
* `main(argv)`：CLI，用 `load_settings` 組真實元件，印出兩個數字。

## 5. 錯誤處理（可靠性）

* 抽取 fail-safe（壞 JSON／LLM 失敗 → 空 `Consolidation`）；單段 embed 失敗 → 跳過。
* 批次屬**非對話路徑**：store 寫入錯誤直接拋出（CLI 報錯即可），不影響線上對話（線上 recall 早已各自 fail-safe）。
* 重用既有自訂例外（`EmbeddingError` 用於 embed 跳過）。

---

## 6. 測試策略（全本機，不需網路）

* `test_longterm_extractor`：`_parse_consolidation` 解析合併 JSON（facts＋episodes）；壞 JSON→空；未知 enum 回退；clamp；跳過缺 content；fake LLM 整合；空 messages／LLM 失敗→空。
* `test_longterm_consolidation`：`run_consolidation` 用 fakes（stub 短期、fake extractor、flaky embedder、in-mem fact/vector store）→ 路由正確、計數正確、embed 失敗跳過。
* 移除上述 4 個測試檔後，`uv run pytest` 仍全綠（無其它引用）。

## 7. 已定預設值（列出供否決）

* 合併 JSON 格式：`{"facts": [...], "episodes": [...]}`。
