# Mem0 新演算法對齊與清除死 Neo4j 設定 設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶／資料層（長期記憶骨幹）。
> 修正前置：[記憶層雲端遷移](2026-06-29-記憶層雲端遷移-design.md)（該案假設 Mem0-g graph 模式可用；本案依實測修正）。
> 借鏡來源：mem0 原始碼（`mem0ai` 2.0.10，本機 `temp/` 參考、不進版控）。
> 本案為「Hermes / Mem0 借鏡與優化」系列第 3 項（MEM-UP）。

---

## 1. 背景與目標（實測修正）

「記憶層雲端遷移」案假設採用 **Mem0-g（graph 模式）＝ Supabase pgvector ＋ Neo4j Aura**。實測 mem0 2.0.10 原始碼（=PyPI 最新版、=已安裝版）後**確認此假設不成立**：

* `MemoryConfig`（`mem0/configs/base.py`）只有 7 個欄位：`vector_store / llm / embedder / history_db_path / reranker / version / custom_instructions`，**沒有 `graph_store`**。
* `Memory.from_config` 以 `MemoryConfig(**config_dict)` 建構，pydantic 預設**忽略未知欄位** → [mem0_factory.py](../../../src/kinsun/mem0_factory.py) 傳的 `graph_store: {neo4j}` 一直被**靜默丟棄**。
* 整包 `mem0/` 無 Neo4j graph 程式（`graph_store/neo4j/enable_graph` 僅見於 `exceptions.py`，且綁選用套件 `kuzu`，非 Neo4j Aura）。
* 結論：**`graph_store` 設定、`neo4j` 依賴、`NEO4J_*` 環境變數皆為死重量**；而 `load_settings` 還以 `_require` 強制要求 `NEO4J_*` 才能啟動（卡死金鑰）。
* 另一面：`MemoryConfig.version` 預設 `"v1.1"` → mem0 的**新演算法**（單次抽取、entity linking、語意＋BM25＋entity 多訊號檢索）**已自動生效**。

**本案目標：** 清除死的 Neo4j 設定/依賴/環境變數，並把「採用新演算法」的意圖顯式釘住（`version="v1.1"`）。

**完成後：** 啟動不再要求 Neo4j 金鑰；config 不含死設定；新演算法輸出格式被顯式鎖定。長期記憶的「關係感知」交由 mem0 內建 entity linking + 多訊號檢索（呼叫 `search` 自動生效）。

## 3. 變更內容（逐檔）

### 3.1 mem0_factory.py
移除：
```python
        "graph_store": {
            "provider": "neo4j",
            "config": {
                "url": settings.neo4j_uri,
                "username": settings.neo4j_username,
                "password": settings.neo4j_password,
            },
        },
```
於 `vector_store` 之後（`custom_instructions` 之外）新增：
```python
        "version": "v1.1",
```
`from kinsun.longterm import provenance` 保留（`custom_instructions` 仍需）。移除 `graph_store` 後檔案不再引用 `settings.neo4j_*`。

### 3.2 config.py
`Settings` 移除三欄位 `neo4j_uri / neo4j_username / neo4j_password`；`load_settings` 移除：
```python
        neo4j_uri=_require(env, "NEO4J_URI"),
        neo4j_username=_require(env, "NEO4J_USERNAME"),
        neo4j_password=_require(env, "NEO4J_PASSWORD"),
```
`DATABASE_URL` 仍 `_require`（向量庫＋短期＋帳號共用，必填）。

### 3.3 pyproject.toml / uv.lock
`uv remove neo4j`。確認無 `src/` 程式 `import neo4j`（已查證：無）。

### 3.4 .env.example
移除：
```dotenv
NEO4J_URI=neo4j+s://xxxx.databases.neo4j.io
NEO4J_USERNAME=neo4j
NEO4J_PASSWORD=
```

### 3.5 living docs
* `README.md`「啟動本機開發」必填清單移除 Neo4j Aura 段，保留 LINE/Gemini/Supabase。
* `progress.md` 技術棧表「長期記憶」列由「Mem0（graph）＝pgvector＋Neo4j Aura」改為「Mem0 v1.1 新演算法（單次抽取＋entity linking＋多訊號檢索），向量庫 Supabase pgvector」；移除啟動警告中的 `NEO4J_*`。

## 4. 錯誤處理 / 相容性

* 移除死設定不改任何讀寫控制流；`Mem0LongTermStore` 的 `add/search` fail-safe 全保留。
* 移除 `NEO4J_*` 必填 → 既有以 `BASE_ENV` 帶 `NEO4J_*` 的測試在「未知 key 被忽略」下仍可運作，但本案一併清除以反映現實。
* `version="v1.1"` 為 `MemoryConfig` 合法值（即現行預設），零行為改變、僅鎖定意圖。

## 6. 已知取捨（列出供否決）

* **全清 neo4j 依賴/env**：未來自建 Neo4j 關係層需加回；已採「乾淨優先」。
* **顯式釘 `version="v1.1"`**：防 mem0 未來預設漂移；代價是未來想升 v2 格式需手動改。小專案「穩定可預期」優先。
* **不回改歷史 plan/spec**：保留決策存底；以本 spec 作修正紀錄並交叉引用。
* **真正新演算法效果待雲端驗證**：本案僅改設定/組裝，離線單元測涵蓋「設定正確」；實機效果待金鑰冒煙。
