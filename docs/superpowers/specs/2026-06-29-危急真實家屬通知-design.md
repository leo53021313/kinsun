# 危急真實家屬通知 v1 設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：③ 安全／危急、② 後端服務、⑦ 資料層。
> 策略文件：[危急偵測與誤報處理設計](../../危急偵測與誤報處理設計.md)（本 spec 實作其 §7 升級鏈的**第一刀**）。
> 前置模組：[危急偵測核心](2026-06-29-危急偵測核心-design.md)、[帳號綁定核心](2026-06-29-帳號綁定核心-design.md)、[記憶層雲端遷移](2026-06-29-記憶層雲端遷移-design.md)。

---

## 1. 背景與目標

危急偵測核心已能在 `tier ≥ L2` 時呼叫 `Notifier.notify(session_id, assessment)`，但目前掛的是 `LogNotifier`（placeholder，只寫 log，不真的通知家屬）。帳號綁定核心已落地（`PgAccountRepository` + `AccountService`，含 `guardians_of`、escalation 順序），「長者—家屬關聯」這份資料現在已存在。

本 spec 把 placeholder 換成**真的推播給家屬**：偵測到危急 → 由綁定資料查出該長者的家屬 → 依升級順序逐一 LINE push 一則危急提醒。

**完成後：** 已綁定的長者一旦觸發 `L2/L3`，其所有家屬（依 `escalation_order`）會收到一則含風險原因與免責提醒的 LINE 訊息。未綁定或查無家屬 → fail-safe 記錄、不中斷對話、不崩潰。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* 儲存層查詢：`get_elder_by_line(line_user_id)`、`get_guardian(guardian_id)`（`AccountRepository` Protocol + `PgAccountRepository` + 測試用 `FakeAccountRepository`）。
* 綁定服務：`AccountService.guardian_line_ids(elder_line_id) -> list[str]`（長者 LINE id → 各家屬 LINE id，依 escalation 順序）。
* 新通知器：`LineGuardianNotifier`（實作既有 `Notifier` Protocol），逐一 push 給所有家屬；單一失敗隔離。
* 訊息格式 `_format_alert`（含風險原因、等級、免責；`L3` 追加自行評估 119）。
* 組裝：`app.py` 以 `LineGuardianNotifier` 取代 `LogNotifier`。

### 2.2 不在範圍內（v2／後續）
* **升級鏈與 ack／逾時**（主家屬未回應 → 升級次家屬 → 緊急聯絡人；策略文件 §7 的 T1/T2 計時）。
* **多通道**（簡訊／電話／其他 IM）；本案只走 LINE push。
* **L2／L3 差異化通知對象**（v1 兩者皆推給全部家屬，僅訊息文案差異）。
* **節流／去重**（同一事件短時間多次 `L2+` 會重複推播）。
* **綁定閘門**（對話流程強制綁定才回應）——既有缺口，另案。
* `can_view_transcript` 權限套用到通知內容（v1 訊息不含逐字稿）。

## 3. 資料流

```
LINE 語音 → VoicePipeline.process(session_id = 長者 LINE userId)
  → detector.assess() → tier ≥ L2
    → LineGuardianNotifier.notify(session_id, assessment)
        directory.guardian_line_ids(session_id)            # = AccountService
          ├─ repo.get_elder_by_line(session_id) → Elder | None
          │     └─ None（未綁定）→ 回 []
          └─ repo.list_elder_guardians(elder_id)（依 escalation_order）
                └─ 逐筆 repo.get_guardian(guardian_id) → 收集 line_user_id
        targets == []  → logger.warning（查無家屬）、return（fail-safe）
        targets 非空   → text = _format_alert(assessment)
                         for line_id in targets: pusher.push_text(line_id, text)（單一失敗隔離）
```

`session_id` 即 webhook 取得的長者 LINE `userId`（與短期記憶／Mem0 的 key 一致）。家屬清單只含 `guardians`（主家屬＋一般家屬），不含長者本人。

## 4. 元件與介面

### 4.1 儲存層（accounts/repository.py、tests/fakes.py）

`AccountRepository` Protocol 新增兩個唯讀查詢，三個實作（Pg／Fake）同步補上：

```
get_elder_by_line(line_user_id: str) -> Elder | None     # SELECT ... WHERE line_user_id = %s
get_guardian(guardian_id: str)      -> Guardian | None    # SELECT ... WHERE guardian_id = %s
```

* `PgAccountRepository`：參數化查詢，沿用 `_query`／`AccountError`。
* `FakeAccountRepository`：`get_elder_by_line` 線性掃 `elders.values()`；`get_guardian` 直接查 `guardians` dict。

> 註：長者 `line_user_id` 未建唯一索引（一位長者一個 LINE，綁定流程保證）；v1 取第一筆即可。

### 4.2 綁定服務（accounts/service.py）

```python
def guardian_line_ids(self, elder_line_id: str) -> list[str]:
    elder = self._repo.get_elder_by_line(elder_line_id)
    if elder is None:
        return []
    line_ids = []
    for eg in self._repo.list_elder_guardians(elder.elder_id):   # 已依 escalation_order 排序
        guardian = self._repo.get_guardian(eg.guardian_id)
        if guardian is not None and guardian.line_user_id:
            line_ids.append(guardian.line_user_id)
    return line_ids
```

唯讀、無副作用；查無長者或無家屬 → `[]`。

### 4.3 通知器（safety/notifier.py）

保留 `Notifier` Protocol 與 `LogNotifier`（仍有測試／可作 fallback）。新增：

```python
class GuardianDirectory(Protocol):
    def guardian_line_ids(self, elder_line_id: str) -> list[str]: ...

class Pusher(Protocol):
    def push_text(self, user_id: str, text: str) -> None: ...

class LineGuardianNotifier:
    def __init__(self, directory: GuardianDirectory, pusher: Pusher) -> None: ...
    def notify(self, session_id: str, assessment: RiskAssessment) -> None:
        targets = self._directory.guardian_line_ids(session_id)
        if not targets:
            logger.warning("危急但查無可通知家屬 session=%s tier=%s reason=%s", ...)
            return
        text = _format_alert(assessment)
        sent = 0
        for line_id in targets:
            try:
                self._pusher.push_text(line_id, text); sent += 1
            except Exception:  # noqa: BLE001
                logger.exception("推播家屬失敗 line_id=%s", line_id)
        logger.warning("已通知家屬 session=%s tier=%s 成功=%d/%d", ...)
```

**解耦**：`LineGuardianNotifier` 只依賴兩個小 Protocol（`GuardianDirectory`、`Pusher`），**不 import `accounts`／`channels`**。`AccountService`（duck-typed 滿足 `GuardianDirectory`）與 `LineApiMessenger`（滿足 `Pusher`）由 `app.py` 注入。

### 4.4 訊息格式（_format_alert）

```
⚠️【金孫關懷提醒】您關心的長輩可能需要您的注意：{reason}（風險等級 {tier.name}）。請盡快主動關心一下。
（tier == L3 追加）（如情況緊急，請自行評估是否撥打 119。金孫不提供醫療診斷。）
```

呼應策略文件 §9：不替代專業判斷、不下醫療診斷、`L3` 提示家屬自行評估 119。

### 4.5 組裝（app.py）

```python
accounts = AccountService(
    PgAccountRepository(settings.database_url),
    clock=lambda: datetime.now(tz),
    ttl_hours=settings.invite_ttl_hours,
    max_attempts=settings.invite_max_attempts,
)
messenger = LineApiMessenger(settings.line_channel_access_token)   # 移到 pipeline 之前
pipeline = VoicePipeline(..., notifier=LineGuardianNotifier(accounts, messenger))
```

`ensure_schema` 已建帳號 5 表（[db.py](../../../src/kinsun/db.py)），無新增資料表、無新環境變數、無新依賴。

## 5. 錯誤處理（fail-safe）

* **未綁定／無家屬**：`guardian_line_ids` 回 `[]` → 記錄、return，不推播、不拋錯。
* **單一家屬 push 失敗**：`try/except` 隔離，繼續推其餘家屬；最後記錄成功筆數。
* **查詢 DB 失敗**：`PgAccountRepository` 拋 `AccountError`；`guardian_line_ids` 不另吞（屬 `notify` 內，最終仍在 pipeline 的危急分支）。
  > 取捨：`notify` 失敗不應中斷對話回覆。但 `pipeline.process` 先 `agent.handle` 產生回覆、再 `notify`，且通知在 `return tts` 之前 —— 為確保「通知失敗不吃掉長者回覆」，`LineGuardianNotifier.notify` 對**整體**亦不可上拋。**決策：`notify` 內最外層再包一層保護，任何例外只記錄不拋**（與 `LogNotifier` 同為「通知是旁路、不得影響主對話」）。
* secret 全走環境變數；查詢全參數化（防 SQL injection）。

## 6. 測試策略（單元全離線，整合 opt-in）

* `tests/test_accounts_service.py`（`FakeAccountRepository`，注入 clock/id/code）：
  * `guardian_line_ids`：建長者→綁長者 LINE→兌換一名家屬→回 `["U-son", "U-daughter"]`（依序、含主家屬）。
  * 未綁定（查無長者）→ `[]`；長者已綁定但無其他家屬 → `["U-son"]`。
* `tests/test_safety_notifier.py`（fake directory + spy pusher）：
  * 多家屬 → 依序對每位 `push_text`，內容＝`_format_alert`。
  * 空清單 → 不 push、僅記錄 warning。
  * `L3` 訊息含「119」與「不提供醫療診斷」；`L2` 不含 119。
  * 單一 `push_text` 拋例外 → 其餘家屬仍收到（隔離）、`notify` 不上拋。
  * `directory.guardian_line_ids` 拋例外 → `notify` 不上拋（最外層保護）。
* `tests/test_pg_account_repository.py`（opt-in `KINSUN_IT=1`）：`get_elder_by_line`／`get_guardian` round-trip。
* 既有 `test_pipeline`（`SpyNotifier`）、`test_webhook`（`_NullNotifier`）、`LogNotifier` 測試不受影響。

## 7. 已知取捨（列出供否決）

* **v1 推給全部家屬、無升級鏈**：簡單可靠；代價是次／緊急聯絡人會與主家屬同時收到（非「主未回才升級」）。升級鏈為 v2。
* **無節流**：同一危急情境若連續多輪 `L2+`，家屬會重複收到。可接受（寧可多報不可漏報）；節流為 v2。
* **L2 與 L3 同樣推給全部家屬**：只有文案差異；差異化通知對象為 v2。
* **`notify` 全程不上拋**：通知是對話旁路，任何失敗都不得影響長者端回覆（與既有 fail-safe 原則一致）。
* **長者 `line_user_id` 取第一筆**：綁定流程保證一對一，不另建唯一索引。
