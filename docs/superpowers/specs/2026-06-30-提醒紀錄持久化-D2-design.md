# 提醒紀錄持久化設計（D 第二切片 / D2）

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：⑦ 記憶/資料層、⑥ 提醒引擎、② 後端（報告資料源）。
> 前置：[用藥提醒](2026-06-29-用藥提醒-design.md)、[回診提醒](2026-06-30-回診提醒-design.md)（兩個 job 的推播點）、[危急事件持久化 D1](2026-06-30-危急事件持久化-D1-design.md)（同為報告資料流）。
> 本案是「D 報告資料收集」的第二條資料流：提醒紀錄。後續：D3 對話摘要；E 報告檢視讀取 D1+D2+D3。

---

## 1. 背景與目標

用藥/回診提醒目前推出去就沒留痕跡。本案在兩個提醒 job 的推播點落一筆「提醒紀錄」（哪位長輩、哪種、內容、時間），作為健康報告（E）的資料源之一。

**完成後：** 每當實際送出一次提醒（用藥：已同意長輩；回診：事件觸發）即寫一筆 `reminder_logs`；記錄失敗只 log、不影響推播。

**本案不含**：報告 REST/前端（E）；危急事件（D1，已完成）、對話摘要（D3）。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* `src/kinsun/reports/`（新套件）+ `reports/reminders.py`：`ReminderLog`、`ReminderLogError`、`ReminderLogStore` Protocol、`PgReminderLogStore`、`safe_record` 幫手。
* `db.py`：`reminder_logs` 表 DDL + `ensure_schema`。
* `medication/jobs.py`、`appointment/jobs.py`：各加可選 `record` callable，在推播點 `safe_record`。
* `scheduler/worker.py`：建 `PgReminderLogStore`、把 `record` 接進用藥/回診 job。
* `tests/fakes.py`：`FakeReminderLogStore`。測試：med/appt jobs 記錄案例、`test_reminder_logs.py`（PG round-trip opt-in）。

### 2.2 不在範圍內（後續）
* 報告 REST/前端（E）。
* 對話摘要（D3）。
* 「是否送達」回執、每收件人一筆、提醒清理/保留策略。

* DDL：
  ```sql
  CREATE TABLE IF NOT EXISTS reminder_logs (
    log_id TEXT PRIMARY KEY, elder_id TEXT NOT NULL,
    kind TEXT NOT NULL, content TEXT NOT NULL, created_at DOUBLE PRECISION NOT NULL);
  CREATE INDEX IF NOT EXISTS idx_reminder_logs_elder_created

  ```

## 4. job 接點

**用藥 [medication/jobs.py](../../../src/kinsun/medication/jobs.py)**：
* `build_medication_slot_job(..., record: Callable[[str, str, str], None] | None = None)`。
* action 在 `push(...)` 之後：
  ```python
  safe_record(record, elder_id, "medication", f"{label}用藥：{'、'.join(names)}")
  ```
  （`label = SLOT_LABELS[slot]`；未同意/未綁定者在前面已 return，不會記。）

**回診 [appointment/jobs.py](../../../src/kinsun/appointment/jobs.py)**：
* `build_appointment_reminder_job(..., record: Callable[[str, str, str], None] | None = None)`。
* action 在推完長輩/家屬後：
  ```python
  safe_record(record, appt.elder_id, "appointment", f"{when_word}回診：{appt.label}")
  ```
  （事件觸發即記一筆，符合「以長輩為鍵、每次提醒一筆」。）

## 6. 測試策略

**`tests/test_medication_jobs.py`**：`_job` 多收 `record`；
* 已同意長輩推送後 → `record` 收到 `("e1", "medication", "早上用藥：…")`；
* 未同意長輩 → 不記錄。

**`tests/test_appointment_jobs.py`**：
* 事件觸發後 → `record` 收到 `("e1", "appointment", "今天回診：…")`（每事件一筆）。

**`tests/test_reminder_logs.py`（opt-in `KINSUN_IT=1`）**：
* `PgReminderLogStore` record medication + appointment → `list_for_elder` 依 `created_at` desc。

**`tests/fakes.py`**：`FakeReminderLogStore`（`record` 記於 `recorded`；`list_for_elder` 回對應 `ReminderLog`）。

既有不傳 `record` 的 job 測試維持綠（`record=None` 不記）。

---

## 7. 錯誤處理（fail-safe）

* `safe_record` 吞例外、只 log：提醒已送出，記錄失敗不影響推播與其他長輩（fanout 本就逐筆隔離）。
* 查詢失敗 → `ReminderLogError`（日後 E 處理）。

---

## 8. 已定決策（列出供否決）

* 切片＝提醒紀錄（用藥 + 回診一併）；對話摘要為下一條資料流。
* 每次提醒一筆、以 `elder_id` 為鍵（非每收件人）。
* 記錄代表「實際送出」：用藥＝已同意長輩推完後；回診＝事件觸發。
* `record` 以可選 callable 注入 job（比照 `push`）；`reports/` 套件為報告資料中性家。
* 欄位：`kind`/`content`（人類字串）/`created_at`；不存收件人/回執。
