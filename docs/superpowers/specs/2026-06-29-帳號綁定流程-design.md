# 帳號綁定流程 v1（LINE 引導式對話入口）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：① LINE 通道、② 後端服務、⑦ 資料層。
> 前置：[帳號綁定核心](2026-06-29-帳號綁定核心-design.md)（`AccountService`，本 spec 為其使用者入口）、[記憶層雲端遷移](2026-06-29-記憶層雲端遷移-design.md)（Postgres）。
> 策略文件：[帳號綁定與知情同意設計](../../帳號綁定與知情同意設計.md)。

---

## 1. 背景與目標

`AccountService`（建檔／邀請／兌換／同意）已完成，但**沒有任何使用者入口**——webhook 目前只處理語音、文字一律回「只聽得懂語音」。因此沒有人能真的綁定，剛上線的危急家屬通知與未來的用藥提醒都「打不到人」。

本 spec 做**綁定子系統的使用者入口**：在 LINE 用**引導式對話**（單一關鍵字「設定」+ 數字選單）讓家屬建立長輩檔、取得邀請碼、邀請其他家屬，並讓長輩／家屬貼碼完成綁定與知情同意。

**完成後：** 家屬可在 LINE 自助完成「建檔 → 發碼 → 長輩貼碼綁定 → 邀請其他家屬」，危急通知的「推給所有家屬」才名副其實。

---

## 2. 範圍（Scope）

### 2.1 在範圍內
* webhook 開始處理**文字訊息**，路由到 `BindingFlow`（引導式狀態機）。
* 三個引導流程：① 建立長輩（拿 ELDER 邀請碼）② 邀請其他家屬（拿 GUARDIAN 邀請碼）③ 綁定（貼碼 → 確認 → 兌換 + 知情同意）。
* **貼碼自動辨識**：IDLE 時貼上有效邀請碼直接進確認（長輩免操作選單）。
* 每人未完成流程的**狀態**存 Postgres（`binding_sessions`，behind Protocol + fake）；逾時重置。
* 家屬姓名取 LINE 暱稱（`messenger.display_name`）。

### 2.2 不在範圍內（後續）
* **綁定閘門**（未綁定/未同意者擋對話）——另案。
* LIFF UI、rich menu 圖文選單。
* 代理同意 `PROXY`（需另法取得長者 LINE id）；v1 僅 `SELF`。
* 換手機重綁、解除綁定、邀請碼重發節流。

## 4. 狀態機（binding/flow.py）

### 4.1 狀態

```python
class BindingState(StrEnum):
    MENU = "menu"                 # 已顯示選單，等 1/2/3
    AWAIT_ELDER_NAME = "elder_name"   # 流程1：等長輩稱呼
    AWAIT_ELDER_PICK = "elder_pick"   # 流程2：多位長輩，等選號
    AWAIT_CODE = "code"               # 流程3：等貼碼
    AWAIT_CONFIRM = "confirm"         # 流程3/自動：等「是/否」
```

無 session ＝ IDLE（不存列）。`BindingSession(line_user_id, state, data: dict, updated_at)`，`data` 暫存流程資料（如 `{"code": ...}`、`{"elder_ids": [...]}`）。

### 4.2 進入點（`handle(line_user_id, text) -> str | None`）

文字先 `strip()`。判斷順序：

1. **有未過期 session** → 交給對應狀態處理（含「取消」→ 刪 session、回「已取消」）。
2. **無 session**：
   * `text ∈ 觸發詞 {"設定","綁定","選單"}` → 存 MENU、回主選單文字。
   * `text` 形如邀請碼（`^[A-Za-z0-9_-]{16,}$`）且 `accounts.preview_invite(text)` 找得到 → 進「確認」（見 4.4）。
   * 否則 → **回 `None`**（非綁定互動，webhook 維持既有語音提示）。

> 逾時：載入 session 時若 `now - updated_at > ttl` → 視為 IDLE（刪除），text 重走步驟 2。

### 4.3 主選單與流程 1/2

| 狀態/輸入 | 行為 |
|-----------|------|
| MENU + `1` | 存 AWAIT_ELDER_NAME → 回「請問長輩怎麼稱呼？（例：阿公、王媽媽）」 |
| MENU + `2` | `accounts.elders_managed_by(line)`：0 位→回「您還沒有長輩檔案，請先選 1」+ 重顯選單；1 位→直接發 GUARDIAN 邀請碼（見下）；多位→存 AWAIT_ELDER_PICK(data=elder_ids)、列「請回覆數字選擇長輩：1.阿公 2.…」 |
| MENU + `3` | 存 AWAIT_CODE → 回「請貼上您收到的邀請碼」 |
| MENU + 其他 | 回「請回覆 1、2 或 3」 |
| AWAIT_ELDER_NAME + 名字 | `name=text`；`display=profiles.display_name(line)`；`elder=accounts.create_elder(line, display, name)`；`inv=accounts.generate_invite(elder.elder_id, ELDER)` → 刪 session、回「已建立『name』檔案，您是主要家屬。請把這組綁定碼交給 name，在金孫聊天視窗貼上：\n{code}\n（24 小時內有效）」 |
| AWAIT_ELDER_PICK + 數字 n | 取 `elder_ids[n-1]`；`inv=generate_invite(elder_id, GUARDIAN)` → 刪 session、回邀請碼＋教學；越界→回「請回覆清單中的數字」 |

「發 GUARDIAN 邀請碼」訊息：「這是『name』的家人邀請碼，請把它交給其他家屬，在金孫聊天視窗貼上：\n{code}\n（24 小時內有效）」。

### 4.4 流程 3（綁定 + 知情同意）

* **AWAIT_CODE + code**（或自動辨識）→ `p=accounts.preview_invite(code)`：
  * `None`（查無）→ 回「找不到這組邀請碼，請確認後再貼一次」+ 留在 AWAIT_CODE（自動辨識情境則直接 `None`）。
  * `p.reason` 有值 → 回對應訊息（`used`→已被使用；`expired`→已過期，請家屬重新產生；`too_many_attempts`→嘗試次數過多）→ 刪 session。
  * 有效 → 存 AWAIT_CONFIRM(data={"code":code})、回確認＋**知情同意**：
    * ELDER：「您要綁定為『elder_name』本人嗎？綁定後金孫會記錄您的對話以提供關懷與必要時通知家人。同意請回覆『是』，取消請回覆『否』。」
    * GUARDIAN：「您要成為『elder_name』的家人嗎？同意請回覆『是』，取消請回覆『否』。」
* **AWAIT_CONFIRM + 是/好/確認** → `accounts.redeem_invite(code, line, consent_by=SELF)`：
  * 成功 → 刪 session、回「綁定成功！…」（ELDER：可開始用語音跟金孫聊天；GUARDIAN：危急時會收到關懷通知）。
  * `InviteError(reason)` → 回對應訊息、刪 session。
* **AWAIT_CONFIRM + 否/取消** → 刪 session、回「已取消綁定」。

## 5. 儲存（binding/session.py、db.py）

* `BindingSessionStore`（Protocol）：`get(line_user_id) -> BindingSession | None`、`save(session)`、`delete(line_user_id)`。
* `PgBindingSessionStore(database_url)`：表
  `binding_sessions(line_user_id TEXT PRIMARY KEY, state TEXT NOT NULL, data TEXT NOT NULL, updated_at DOUBLE PRECISION NOT NULL)`；
  `data` 存 `json.dumps(dict)`；upsert `ON CONFLICT (line_user_id) DO UPDATE`；DB 例外包成 `AccountError`（沿用，避免新例外型別）。
* `ensure_schema` 增 `BINDING_DDL`。
* `FakeBindingSessionStore`（純 dict）供單元測試。

## 6. 服務／儲存層補強

### 6.1 accounts/service.py
```python
@dataclass(frozen=True)
class InvitePreview:
    role: InviteRole
    elder_name: str
    reason: str | None        # None=有效；否則 used/expired/too_many_attempts

def preview_invite(self, code) -> InvitePreview | None:   # 查無→None；否則回角色+長輩名+有效性
def elders_managed_by(self, line_user_id) -> list[Elder]: # 此 LINE 為家屬的長輩們
```
`preview_invite` 的有效性檢查與 `redeem_invite` 同規則（used/上限/過期）；`redeem_invite` 仍是實際轉移的唯一真實來源。

### 6.2 accounts/repository.py
`AccountRepository` Protocol + `PgAccountRepository` + `FakeAccountRepository` 加：
`elder_ids_of_guardian(guardian_id: str) -> list[str]`（`SELECT elder_id FROM elder_guardians WHERE guardian_id=%s ORDER BY elder_id`）。

### 6.3 channels/line/messenger.py
`LineMessenger` Protocol + `LineApiMessenger` 加 `display_name(user_id: str) -> str`（LINE `get_profile(user_id).display_name`；失敗回 `""`，不阻斷）。`binding/flow.py` 定義 `Profiles`(Protocol：`display_name`)，由 messenger 滿足（binding 不 import channels）。

## 7. 組裝與路由

* **webhook.py**：`create_app(*, parser, pipeline, messenger, binding)`。`_handle_events` 對 `message.type=="text"`：`reply = binding.handle(session_id, text)`；`messenger.reply_text(reply_token, reply or NON_AUDIO_PROMPT)`。其餘非語音（貼圖/圖片）維持 `NON_AUDIO_PROMPT`。`session_id` 同語音取 `event.source.user_id`。
* **app.py**：沿用已建的 `accounts`（`AccountService`），新增
  `binding = BindingFlow(accounts, PgBindingSessionStore(settings.database_url), messenger, clock=lambda: datetime.now(tz), session_ttl_seconds=settings.binding_session_ttl_minutes*60)`，注入 `create_app`。
* **config.py / .env.example**：`BINDING_SESSION_TTL_MINUTES`（預設 10）。

## 8. 錯誤處理（fail-safe）

* `BindingFlow.handle` **永不上拋**：內部 `try/except`，DB／服務例外 → 回「系統忙線，請稍後再試一次」，必要時刪 session 重來。
* 邀請碼錯誤以 `InvitePreview.reason` / `InviteError.reason` 對應友善訊息（不洩漏內部細節）。
* `display_name` 失敗回 `""`（家屬姓名可空，不阻斷建檔）。
* 非綁定文字 → `None` → webhook 維持既有行為（不劫持一般訊息）。
* secret 走環境變數；查詢全參數化。

## 9. 測試策略（單元全離線）

* `tests/test_binding_flow.py`（注入 `AccountService(FakeAccountRepository)` + `FakeBindingSessionStore` + fake `Profiles`，注入 clock）：
  * 觸發詞 → 主選單；非綁定文字 → `None`。
  * 流程1：1 → 問名 → 建檔 + 回 ELDER 碼（驗 `create_elder`/`generate_invite` 被呼叫、訊息含碼與長輩名、用到 `display_name`）。
  * 流程2：0 位長輩提示；1 位直接發碼；多位 → 選號 → 發 GUARDIAN 碼。
  * 流程3：貼有效碼 → 確認（ELDER/GUARDIAN 文案、含同意字樣）→「是」→ `redeem_invite(consent_by=SELF)` 成功；「否」→ 取消。
  * 無效碼各 reason（查無/已用/過期/上限）；自動辨識（IDLE 貼碼）→ 確認。
  * 「取消」於各狀態 → 重置；逾時 → 重置。

* `tests/test_accounts_service.py`：`preview_invite`（各 reason）、`elders_managed_by`（0/1/多）。

## 10. 已定預設值（列出供否決）

* 觸發詞 `{"設定","綁定","選單"}`；確認 `是/好/確認`、取消 `否/不要/取消`；數字接受半形/全形。
* 邀請碼自動辨識門檻 `^[A-Za-z0-9_-]{16,}$`（對應 `token_urlsafe(12)`）。
* session 逾時 10 分鐘（`BINDING_SESSION_TTL_MINUTES`）。
* `consent_by=SELF`（本人用自己手機操作即本人同意）；`PROXY` 後續。
* 家屬姓名 = LINE 暱稱（取不到則空字串）。
* 一位家屬可管理多位長輩（流程2 以選號處理）。
* 錯誤一律回友善中文、`handle` 不上拋。
