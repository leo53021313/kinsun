# 命名一致性盤點與統一——設計文件

- 日期：2026-07-03
- 分支：Leo
- 狀態：已核可（待實作計畫）

## 背景

專案經過多輪開發後，出現大量「同概念異名」的狀況，例如 `elder_id`（229 處）與 `user_id`（110 處）並存、`jobs.py` 同時存在於 `proactive/` 與 `scheduler/`、資料存取層有 `store.py` 與 `repository.py` 兩種命名等。命名發散提高維護成本，也讓新功能開發時無所適從。

## 目標

1. 全面盤點專案中的命名分歧，產出《命名決策表》供逐項決策。
2. 依核可結果一次性完成改名，統一全專案命名。
3. 將最終命名規範寫入 `AGENTS.md`，避免日後再度發散。

## 範圍與界線

盤點與改動涵蓋以下五個面向，**全部允許改動**：

| 面向 | 內容 | 備註 |
|---|---|---|
| 程式識別字 | 變數、函式、類別、方法的跨模組同概念異名 | |
| 環境變數與設定 | `.env.example`、`config.py`、`services/asr`、`services/tts`、docs 之間的鍵名 | DGX 上實際 `.env` 不在版控，需列出人工同步清單 |
| 檔案與模組命名 | 模組檔名、目錄命名風格 | |
| API 路由與欄位 | `web/api.py` 的路徑、參數、JSON 欄位；`frontend/` 依賴一併更新 | |
| DB Schema | 資料表與欄位名稱 | 開發階段資料可丟，**不寫遷移腳本**，直接改 Schema 重建 |

### 非目標（Non-goals）

- 不做與命名無關的重構、格式調整或行為變更。
- 不寫資料庫遷移腳本。
- 未經核可的命名項目一律不動。

## 流程設計

### 第一階段：盤點

平行派出探索代理，每個面向一個代理，各自產出該面向的分歧清單。彙整為《命名決策表》，存於 `docs/superpowers/specs/2026-07-03-naming-decisions.md`，每項包含：

- 編號
- 面向（識別字／環境變數／檔案模組／API／DB）
- 現況各種寫法與出現處數
- 建議統一名與理由
- 影響範圍（檔案數、是否觸及對外介面）

### 第二階段：決策

決策表交由使用者逐項核可；使用者可回覆「全部照建議」或指定「#N 改成 xxx」。未核可項目不改。

### 第三階段：執行

- **前置**：工作區現有 `src/kinsun/channels/inbound.py` 與 `tests/test_inbound.py` 未提交變更，改名前由使用者決定先提交或暫存（stash），避免混入無關變更。
- 一次性改完，PR 內依類別分五個 commit，順序由低風險到高風險：
  1. 檔案／模組命名
  2. 程式識別字
  3. API 路由與欄位（含 `frontend/` 同步）
  4. 環境變數（含 `.env.example` 與文件同步）
  5. DB Schema
- 每個 commit 完成後執行全測試（pytest），通過才進行下一批。
- 同步更新受影響文件（README、docs、`.env.example`），並將最終命名規範寫入 `AGENTS.md`。
- 依專案規範不自動 push，完成後等使用者指示。

## 驗證方式

- 每批改名後跑全測試，全數通過。
- 以 grep 檢查舊名稱殘留（含字串、註解、文件）。
- 專案可正常建置（`uv sync` 與服務啟動不受影響）。

## 風險與處理

| 風險 | 處理方式 |
|---|---|
| 與其他 6 位協作者的分支衝突 | 單一大 PR；合併前通知整合負責人，建議其他人先合併手上工作 |
| 全域取代誤傷（字串、同名不同義） | 不盲目全域取代，逐處確認語意；以測試與 grep 驗證 |
| DGX 實際 `.env` 與程式碼脫鉤 | 在報告中列出需人工同步修改的環境變數鍵 |
| 決策表項目過多造成決策疲勞 | 決策表附建議名與理由，支援「全部照建議」快速核可 |
