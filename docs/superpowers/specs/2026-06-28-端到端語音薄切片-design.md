# 端到端語音薄切片（Tracer Bullet）設計

> 對應架構圖 [長輩看護系統架構-分層版.drawio](../../長輩看護系統架構-分層版.drawio)：① 前端／通路層（LINE）、③ 語音處理（ASR/TTS）、④ Agent 核心、⑤ 工具／外部（LLM）。
> 本文是金孫 KinSun 的**第一份實作 spec**，定義一條「最薄但端到端打通」的語音管線，作為全團隊共同地基。

---

## 1. 背景與目標

專案目前**設計文件完整、程式碼為零**（[pyproject.toml](../../../pyproject.toml) 的 `dependencies = []`）。7 人協作需要一個共同的程式骨架才能並行開發。

本切片採 **tracer bullet（曳光彈）** 策略：用最薄的實作把一條請求路徑從頭打通，**最早暴露整合風險**、立刻有可 demo 的骨架，之後每位成員再把自己負責的那一段加強為正式實作。

**本切片完成後要能做到：**

長者在 LINE 傳一則**語音訊息** → 系統辨識成文字 → Care Agent 帶著系統指令呼叫 Gemini → 取得繁體國語漢字回應 → 回覆給長者。

## 3. 架構決策（Key Decisions）

### 3.1 位置無關（Location-Independent）：界線是可設定的 endpoint，不是機器

重模型（ASR、未來的 TTS）以**網路服務**形式存在；應用層只是**呼叫端（client）**，透過環境變數設定的 endpoint 連線。

| 情境 | ASR/TTS endpoint 指向 | 程式碼 |
|------|------------------------|--------|
| 在 Windows / macOS 開發 | DGX 遠端 URL，或本機 mock | 不變 |
| 整包搬上 DGX | localhost / 內網 | 不變 |
| 模型在 DGX、應用在別處 | DGX URL | 不變 |

差別只在 `config.py` 讀的環境變數，**不是改架構**。co-locate 或拆開都同一份碼。

### 3.2 跨平台紀律（Cross-Platform Discipline）

開發橫跨 **Windows / macOS / DGX Spark（Linux + ARM64/aarch64）**（DGX 同時是開發、伺服器與部署環境），程式碼從第一行就守：

* **OS-agnostic**：一律 `pathlib`、不寫死路徑與分隔符、設定／金鑰全走環境變數。
* **ARM64 相容**：應用層依賴挑有 `linux/aarch64` wheel 的；吃 GPU 的重模型只跑在 DGX，不要求在 Win/macOS 安裝。
* 用 **uv** 維持跨平台環境一致。

（同步寫入 [AGENTS.md](../../../AGENTS.md) 的「開發與部署環境」章節。）

### 3.3 tracer bullet：介面可替換

ASR、TTS 都先定義**抽象介面**，再提供可替換實作（mock / DGX / placeholder）。這讓：

* 7 人能**並行**——上層對著介面開發，不必等真模型。
* 真模型上線後**零痛感替換**——只換實作類別，呼叫端不動。

## 4. 元件與職責（Components）

```
src/kinsun/
  channels/line/      # LINE webhook：驗簽、收語音、下載音檔、回覆
  agent/              # Care Agent 樞紐：組 prompt → 呼叫 LLM → 產生回覆
  llm/                # Gemini client：送系統指令＋使用者文字，取回應文字
  speech/
    asr.py            # ASRClient 介面 + DgxAsrClient（HTTP）+ MockAsrClient
    tts.py            # TTSClient 介面 + TextBubbleTts（placeholder）
  config.py           # 從環境變數讀 endpoint / 金鑰 / 開關
services/
  asr/                # ★跑在 DGX 的 ASR 推論服務（FastAPI 包 Breeze-ASR-26）
tests/
```

| 元件 | 單一職責 | 依賴 |
|------|----------|------|
| `channels/line` | 接 LINE webhook、驗證簽章、下載語音、把回覆送回 LINE | LINE SDK、`agent` |
| `agent` | 樞紐：用系統指令＋使用者文字組 prompt、呼叫 LLM、回傳回覆文字 | `llm` |
| `llm` | 封裝 Gemini 呼叫，輸入系統指令＋文字，輸出繁體國語漢字 | Gemini SDK |
| `speech/asr` | 定義 `ASRClient`，提供 DGX 版與 mock 版 | （DGX 版）`services/asr` |
| `speech/tts` | 定義 `TTSClient`，提供文字泡泡 placeholder | 無 |
| `config` | 集中讀取環境變數，提供型別化設定物件 | 無 |
| `services/asr` | 在 DGX 上載入 Breeze-ASR-26，提供 `POST /transcribe` | GPU、Breeze-ASR-26 |

## 5. 資料流（Data Flow）

```
長者 ──語音訊息──▶ LINE Platform ──webhook──▶ channels/line
                                                  │ 下載音檔 bytes
                                                  ▼
                                            ASRClient.transcribe(bytes)
                                       （dev: MockAsrClient / 正式: DgxAsrClient → DGX）
                                                  │ 繁體國語漢字 str
                                                  ▼
                                            agent.handle(text)
                                                  │ 組 prompt（系統指令＋文字）
                                                  ▼
                                            llm（Gemini）── 回應文字 ──▶ agent
                                                  │ 繁體國語漢字 str
                                                  ▼
                                            TTSClient.synthesize(text)
                                       （placeholder: 直接回文字，不產音檔）
                                                  │
                                                  ▼
                                  channels/line 回覆 ──▶ LINE Platform ──▶ 長者
```

> 全程文字＝**繁體國語漢字**（呼應架構圖「語言流」）。國語→台語的轉換只發生在未來真 TTS 的聲學層，本切片不處理。

## 6. 介面定義（Interfaces）

兩個核心介面是團隊並行與日後替換的關鍵。**簽章在實作時定稿，以下為設計意圖**：

```python
# speech/asr.py
class ASRClient(Protocol):
    def transcribe(self, audio: bytes, *, content_type: str) -> str:
        """音檔 bytes → 繁體國語漢字字串。失敗時拋出 ASRError。"""

class DgxAsrClient(ASRClient):   # 正式：HTTP 呼叫 DGX 上的 services/asr
    ...
class MockAsrClient(ASRClient):  # dev：回固定／可設定文字，不連網
    ...
```

```python
# speech/tts.py
class TTSClient(Protocol):
    def synthesize(self, text: str) -> TtsResult:
        """國語漢字 → 語音。placeholder 階段 audio 為 None，僅帶文字。"""

class TextBubbleTts(TTSClient):  # placeholder：回文字泡泡，audio=None
    ...
```

`TtsResult` 預留 `audio: bytes | None` 與 `text: str`；上層依 `audio` 是否存在決定回語音或回文字——真 TTS 上線後上層**不需修改**。

集中在 `config.py`，全部來自環境變數，**不寫死任何金鑰**（呼應 [AGENTS.md](../../../AGENTS.md) 安全性）。提供 `.env.example` 範本；真正的 `.env` 不進版控。

## 8. 錯誤處理（Error Handling）

呼應 [AGENTS.md](../../../AGENTS.md)「不要忽略任何錯誤」：

* **輸入驗證**：LINE webhook 一律先**驗章**；非語音訊息回友善提示，不當錯誤。
* **可辨識錯誤型別**：`ASRError`、`LLMError` 等自訂例外，不吞例外。
* **對使用者的 fallback**：ASR 或 LLM 失敗時，回一句長者聽得懂的安撫話術（例：「金孫剛剛沒聽清楚，您可以再說一次嗎？」），**絕不把堆疊或技術訊息丟給長者**。
* **逾時**：對 DGX、Gemini 的呼叫都設逾時，避免 webhook 卡住。
* 本切片日誌維持最小（結構化記錄錯誤即可）；完整 ⑧ 監控層為後續子專案。

## 9. 測試策略（Testing）

依 [AGENTS.md](../../../AGENTS.md) 測試順序：

* **單元測試（不需 GPU、不需網路）**：用 `MockAsrClient` 與 `TextBubbleTts`，驗證 `agent.handle()` 流程、`config` 讀取、LINE 驗簽邏輯。這是 dev 期主力，**Windows / macOS / DGX 三邊都能跑**。
* **整合測試**：`ASR_BACKEND=dgx` 連 DGX 上的 `services/asr` 驗證真實辨識；Gemini 用測試金鑰實打或錄製回應。需 DGX 可達時才跑。
* **人工驗證**：用 LINE 實際傳語音，確認端到端回覆（dev 期 webhook 以內網穿透／ngrok 對外）。

CI（pre-commit / GitHub Actions）只跑**不依賴 GPU 與外部服務**的單元測試。
